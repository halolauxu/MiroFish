## 二、统一本体设计（一次生成，全局复用）

MiroFish 原版每次预测都要调 LLM 生成本体（~6K tokens），批量场景下改为**预定义统一本体**：

### 实体类型（Entity Types）

```yaml
entities:
  # ---- 核心实体 ----
  - Company:          # 上市公司（~5000个）
      attrs: [code, name, market, industry_l1, industry_l2, market_cap, pe, pb]
  - Industry:         # 申万行业（31个一级 + 134个二级）
      attrs: [code, name, level, sector]
  - Concept:          # 概念板块（~300个）
      attrs: [name, description, heat_score]

  # ---- 参与者实体 ----
  - Institution:      # 机构（基金/券商/北向/社保）
      attrs: [name, type, aum, style]
  - Person:           # 关键人物（高管/基金经理/分析师）
      attrs: [name, role, affiliation]

  # ---- 事件与数据实体 ----
  - Policy:           # 政策法规
      attrs: [title, issuer, date, impact_level]
  - Event:            # 市场事件（财报/公告/突发）
      attrs: [title, date, type, sentiment, impact_scope]
  - FinancialReport:  # 财报数据
      attrs: [period, revenue, net_profit, yoy_growth, surprise_pct]

  # ---- 市场状态实体 ----
  - MacroIndicator:   # 宏观指标（PMI/CPI/M2/社融）
      attrs: [name, value, date, trend]
  - PriceAction:      # 量价状态
      attrs: [code, date, close, change_pct, volume, turnover]
```

### 关系类型（Edge Types）

```yaml
edges:
  # ---- 产业链关系 ----
  - SUPPLIES_TO:        Company → Company        # 供应关系
  - COMPETES_WITH:      Company → Company        # 竞争关系
  - COOPERATES_WITH:    Company → Company        # 合作关系
  - BELONGS_TO:         Company → Industry       # 行业归属
  - RELATES_TO_CONCEPT: Company → Concept        # 概念关联

  # ---- 资本关系 ----
  - HOLDS_SHARES:       Institution → Company    # 持股关系（带权重）
  - MANAGES:            Person → Institution     # 管理关系
  - LEADS:              Person → Company         # 高管关系

  # ---- 事件影响关系 ----
  - IMPACTS:            Policy → Industry        # 政策影响行业
  - TRIGGERS:           Event → Company          # 事件触发公司异动
  - PUBLISHED_BY:       FinancialReport → Company # 财报归属
  - CORRELATES_WITH:    MacroIndicator → Industry # 宏观指标关联行业

  # ---- 资金与情绪关系 ----
  - FUND_FLOW:          Industry → Industry      # 行业间资金流向
  - SENTIMENT_SHIFT:    Event → Company          # 事件引发情绪变化
```

## 三、十大策略详细设计

### 策略 1：供应链传导策略

```
触发条件：上游公司发布涨价/缺货/扩产公告
信号逻辑：
  1. 事件检测：监控公告关键词（涨价/调价/限产/扩产）
  2. 图谱遍历：Event → TRIGGERS → Company_A → SUPPLIES_TO → [Company_B, C, D...]
  3. LLM评估：对每个下游公司评估「成本转嫁能力」「库存水位」「替代供应商」
  4. 输出信号：{stock, direction, confidence, transmission_delay_days}

数据需求：
  - 公告全文（巨潮资讯网）
  - 供应链关系（年报披露前五大客户/供应商 + LLM补充）
  - 历史传导案例（回测用）

Token消耗：
  - 事件检测：~1K/公告（关键词匹配为主，LLM仅做确认）
  - 影响评估：~3K/下游公司
  - 典型场景：1个涨价事件 × 10个下游 = ~31K tokens

图谱构建：
  - 初始化：LLM解析年报前五大客户/供应商 → 写入 SUPPLIES_TO 边
  - 增量更新：每季财报更新一次
  - 覆盖范围：先做重点行业（锂电/半导体/化工），~500家公司
```

### 策略 2：股东/机构关联策略

```
触发条件：季报/月报披露机构持仓变动
信号逻辑：
  1. 持仓解析：提取基金季报持仓数据
  2. 图谱分析：Institution → HOLDS_SHARES → [Company群]
  3. 聚类发现：同一机构重仓股中，已涨vs未涨标的
  4. 补涨筛选：未涨标的中基本面无恶化的 → 补涨候选
  5. 输出信号：{stock, peer_stocks_avg_return, gap_pct, confidence}

数据需求：
  - 基金季报持仓（Tushare fund_portfolio）
  - 北向资金每日持仓（AkShare）
  - 十大流通股东（Tushare top10_holders）

Token消耗：
  - 极低，主要是图谱遍历 + 简单规则
  - LLM仅在异常情况下做判断：~1K/股
  - 全A股扫描：~5K tokens（批量SQL查询为主）

图谱构建：
  - 每季度更新 HOLDS_SHARES 边（权重=持仓比例）
  - 北向资金每日更新
  - 无需LLM，纯结构化数据导入
```

### 策略 3：事件传播策略

```
触发条件：重大政策发布 / 行业事件
信号逻辑：
  1. 事件识别：新闻/政策文本 → LLM提取事件实体和影响方向
  2. 一级传播：Policy → IMPACTS → Industry_A → BELONGS_TO → [直接受益公司]
  3. 二级传播：Company_直接受益 → SUPPLIES_TO / COOPERATES → [间接受益公司]
  4. 时间差利用：一级已反应（涨停） → 二级尚未反应 → 买入二级
  5. ReACT推理：对二级受益公司做深度分析（受益程度/确定性）
  6. 输出信号：{stock, propagation_level, delay_estimate, confidence}

数据需求：
  - 政策全文（国务院/发改委/工信部官网）
  - 行业新闻（财联社/东方财富）
  - 行业-公司映射（申万分类）

Token消耗：
  - 事件提取：~2K/事件
  - ReACT深度分析：~5K/候选公司
  - 典型场景：1个政策 × 5个二级受益 = ~27K tokens

图谱构建：
  - Policy → IMPACTS → Industry：LLM解析政策文本（每个政策~3K tokens）
  - Industry → Company 映射：结构化数据，无需LLM
```

### 策略 4：财报预期差策略

```
触发条件：上市公司发布季报/年报/业绩预告
信号逻辑：
  1. 财报解读：LLM提取关键指标 + 边际变化 + 管理层措辞
  2. 预期对比：vs 卖方一致预期（Wind/东财一致预期）
  3. 预期差计算：(实际 - 预期) / 预期 = surprise_pct
  4. 信号生成：
     - 正向预期差 > 10% + 股价未反应 → 买入
     - 负向预期差 < -10% + 股价未跌 → 卖出
  5. 图谱增强：检查同行业其他公司是否也有类似趋势
  6. 输出：{stock, surprise_pct, key_driver, industry_trend, confidence}

数据需求：
  - 财报原文（巨潮PDF → 文本提取）
  - 一致预期数据（Wind/Choice/Tushare）
  - 历史财报（时间序列对比）

Token消耗：
  - 财报解读：~5K/公司（提取关键指标+判断）
  - 批量处理：财报季约800家/周 × 5K = ~4M tokens/周 ≈ ¥4（deepseek）
  - 优化：先用规则筛选预期差>5%的，再用LLM深度分析 → 降至~500K/周

ReACT工具替换：
  - InsightForge → 财报数据检索（营收/利润/毛利率趋势）
  - PanoramaSearch → 同行业横向对比
  - Interview → 研报观点提取
```

### 策略 5：研报观点聚合策略

```
触发条件：定期扫描（每周）/ 事件驱动
信号逻辑：
  1. 研报采集：爬取主流券商研报（标题+摘要+评级+目标价）
  2. LLM提取：标准化为 {analyst, broker, rating, target_price, key_points}
  3. 分歧度计算：
     - 评级分歧：std(rating_score) across brokers
     - 价格分歧：std(target_price) / mean(target_price)
  4. 信号类型：
     - 分歧收敛（空翻多）→ 强烈买入信号
     - 分歧发散 → 观望/做波段
     - 一致看多但股价未涨 → 关注催化剂
  5. 输出：{stock, consensus_rating, divergence_score, recent_upgrades, confidence}

数据需求：
  - 券商研报摘要（东方财富研报中心 / Tushare report_rc）
  - 一致预期评级变动

Token消耗：
  - 研报提取：~1K/篇研报
  - 覆盖Top1000股 × 平均5篇/周 = ~5M tokens/周 ≈ ¥5
  - 优化：仅提取结构化字段，不做深度分析 → 降至~2M/周
```

### 策略 6：公告语义变化策略

```
触发条件：每日公告发布
信号逻辑：
  1. 公告采集：每日抓取全A股公告标题
  2. 关键公告筛选：规则过滤（排除例行公告）
  3. 语义分析：LLM对关键公告做情感/措辞分析
  4. 时序对比：与该公司历史公告语义对比
     - Zep存储历史公告作为Episode → 自动建立时序图谱
     - 检测措辞变化：「稳健增长」→「承压」→「显著改善」
  5. 信号生成：
     - 语义拐点（负→正）+ 股价未反应 → 买入
     - 语义持续恶化 + 市场忽视 → 卖出/回避
  6. 输出：{stock, sentiment_score, sentiment_change, key_phrases, confidence}

数据需求：
  - 公告全文（巨潮资讯网 API）
  - 历史公告存档（Zep Episode）

Token消耗：
  - 公告筛选：~0.5K/公告（关键词初筛后LLM确认）
  - 语义分析：~2K/重要公告
  - 每日约200条重要公告 × 2K = ~400K tokens/日 ≈ ¥0.4

图谱构建：
  - 每条重要公告作为 Episode 存入 Zep
  - 自动与 Company 节点关联
  - 随时间积累形成语义时序
```

### 策略 7：图谱增强多因子策略

```
触发条件：每日/每周定期计算
信号逻辑：
  1. 传统因子计算（无需LLM）：
     - 量价因子：动量/反转/波动率/换手率
     - 财务因子：ROE/营收增速/毛利率变化
     - 估值因子：PE/PB/PS分位数
  2. 图谱因子计算（Zep图谱遍历，无需LLM）：
     - 供应链中心度：PageRank(SUPPLIES_TO图)
     - 机构聚集度：count(distinct Institution → HOLDS_SHARES → Company)
     - 概念热度：sum(Concept.heat_score for related concepts)
     - 事件暴露度：count(recent Event → TRIGGERS → Company)
     - 行业龙头度：Company.market_cap / sum(Industry_peers.market_cap)
     - 关联涨跌差：avg(peer_returns) - self_return（补涨/补跌信号）
  3. 因子合成：
     - 等权/IC加权合成
     - 机器学习合成（XGBoost/LightGBM）
  4. 输出：{stock, composite_score, top_factors, rank}

数据需求：
  - 日频行情（AkShare/Tushare）
  - 财务数据（季频）
  - 图谱拓扑（Zep，已有）

Token消耗：
  - 极低！核心计算全在本地
  - LLM仅用于因子解释（可选）：~1K/因子
  - 全A股日频运行：~0 tokens（纯数值计算）

核心价值：
  - 将图谱关系量化为可回测的因子
  - 与传统量化框架（Freqtrade）无缝对接
  - 最容易验证alpha的策略
```

### 策略 8：行业轮动策略

```
触发条件：每周末/重大事件后
信号逻辑：
  1. 行业表现扫描：31个申万一级行业的周涨跌幅/资金流向
  2. 宏观环境分析：LLM分析本周宏观数据（PMI/CPI/政策会议）
  3. 图谱推理：
     - MacroIndicator → CORRELATES_WITH → Industry（宏观驱动）
     - Industry_A → FUND_FLOW → Industry_B（资金流向）
     - Policy → IMPACTS → Industry（政策驱动）
  4. ReACT多跳推理：
     - 当前利好行业 → 产业链上下游 → 下一轮受益行业
     - 历史轮动模式匹配
  5. 输出：{industry, direction, catalyst, timing, confidence}

数据需求：
  - 行业指数行情（申万行业指数）
  - 行业资金流向（北向/融资融券/主力资金）
  - 宏观数据日历（PMI/CPI/LPR/社融）

Token消耗：
  - 宏观分析：~5K/周
  - 行业推理：~3K × 5个重点行业 = ~15K/周
  - 月消耗：~80K tokens ≈ ¥0.08

图谱构建：
  - 行业间 FUND_FLOW 边：每周更新（资金流向数据）
  - MacroIndicator 节点：每月更新
  - Policy → Industry 边：事件驱动更新
```

### 策略 9：产业链景气度传导策略

```
触发条件：月度数据更新（PMI/产量/价格指数）
信号逻辑：
  1. 景气指标采集：各行业PMI/产量/价格/库存数据
  2. 景气度评分：LLM综合评估行业景气度（0-100分）
  3. 传导模型：
     - 上游景气↑ → 图谱遍历 → 中游（滞后1-3月）→ 下游（滞后3-6月）
     - 传导强度 = f(供应集中度, 成本占比, 替代性)
  4. 领先信号：
     - 上游景气拐点 → 预测下游3-6个月后的景气变化
     - 找到景气即将改善但股价未反应的下游行业/公司
  5. 输出：{industry, prosperity_score, trend, leading_signal, target_companies}

数据需求：
  - 行业PMI/子行业PMI（统计局/财新）
  - 大宗商品价格（南华指数/LME/螺纹钢）
  - 行业产量/库存数据（统计局月度数据）

Token消耗：
  - 景气评估：~3K × 31行业 = ~93K/月
  - 传导分析：~5K × 10个重点产业链 = ~50K/月
  - 月消耗：~143K tokens ≈ ¥0.14

图谱构建：
  - Industry → Industry 的 SUPPLIES_TO 边 + 传导权重
  - MacroIndicator 时序节点（每月快照）
```

### 策略 10：市场情绪仿真策略（OASIS）

```
触发条件：重大事件（暴雷/重组/政策突变）→ 手动触发
信号逻辑：
  1. 事件注入：将真实事件作为仿真初始帖子
  2. Agent设定：基于图谱中的 Person/Institution 生成Agent
     - 散户（跟风/恐慌）、机构（理性/调仓）、游资（博弈/打板）
  3. OASIS仿真：模拟不同角色对事件的反应和传播
  4. 情绪聚合：统计仿真结果中的多空比例、情绪强度
  5. 对比现实：仿真预测 vs 实际市场反应 → 寻找定价偏差
  6. 输出：{stock, simulated_sentiment, simulated_price_impact, real_gap, opportunity}
```

```python
# freqtrade strategy 对接示例
class GraphEnhancedStrategy(IStrategy):
    def populate_indicators(self, dataframe, metadata):
        code = metadata['pair'].replace('/', '.')  # e.g., "600519.SH"

        # 读取图谱因子（策略7）
        graph_factors = self.dp.get_graph_factors(code)
        dataframe['supply_chain_centrality'] = graph_factors['pagerank']
        dataframe['institution_concentration'] = graph_factors['inst_count']

        # 读取LLM信号（策略1-6, 8-10）
        signals = self.dp.get_strategy_signals(code)
        dataframe['llm_signal'] = signals['composite_direction']
        dataframe['llm_confidence'] = signals['composite_confidence']

        return dataframe

    def populate_entry_trend(self, dataframe, metadata):
        dataframe.loc[
            (dataframe['llm_signal'] == 'long') &
            (dataframe['llm_confidence'] > 0.7) &
            (dataframe['supply_chain_centrality'] > 0.5),
            'enter_long'] = 1
        return dataframe
```
